🌟1.1 Les 6 Avantages du Cloud AWS

1️⃣ Échanger CapEx contre OpEx

CapEx (Capital Expenditure) :
  • Investissement initial massif en serveurs, datacenters
  • Amortissement sur 3-5 ans
  • Coûts fixes élevés même si sous-utilisé
  • Exemple : 100 000€ pour un datacenter on-premise
OpEx (Operational Expenditure) :
  • Paiement à l'utilisation mensuelle
  • Coûts variables selon consommation réelle
  • Zéro investissement initial
  • Exemple : 500€/mois selon utilisation

✅ Avantage : Flexibilité financière maximale, pas d'immobilisation de capital

2️⃣ Économies d'échelle massives

Comment ça marche :

AWS regroupe l'utilisation de centaines de milliers de clients → achats en très gros volume → prix unitaires réduits répercutés aux clients

  • 67 baisses de prix depuis 2006
  • Négociations favorables avec fournisseurs hardware
  • Innovation continue (processeurs Graviton, stockage custom)
  • Datacenters ultra-optimisés (PUE < 1.2 vs 2.0 moyenne industrie)

✅ Résultat : Vous payez moins cher qu'en achetant vous-même

3️⃣ Ne plus deviner la capacité

Problème on-premise :
Scénario Conséquence
❌ Sous-provisionnement Performances dégradées, clients insatisfaits, perte de revenus
❌ Sur-provisionnement Ressources inutilisées, argent gaspillé, capacité dormante
✅ Solution AWS :
  • Auto Scaling : ajuste automatiquement selon demande réelle
  • Scale up/down en quelques minutes
  • Payez uniquement la capacité utilisée
  • Testez différentes configurations facilement

4️⃣ Augmenter la vitesse et l'agilité

Action On-Premise AWS Cloud
Nouveau serveur Semaines/Mois ⚡ Minutes
Environnement de test Difficile, coûteux ✅ Facile, peu coûteux
Expérimentation Risquée financièrement ✅ Faible coût, arrêt facile
Mise à l'échelle Acheter + installer hardware ⚡ Clic ou automatique

✅ Impact : Innovation accélérée, Time-to-market réduit, Tests A/B sans risque

5️⃣ Arrêter de dépenser pour l'exploitation datacenter

Coûts cachés d'un datacenter on-premise :
  • 💰 Électricité 24/7 (serveurs + climatisation)
  • 💰 Refroidissement continu
  • 💰 Espace physique (loyer ou immobilier)
  • 💰 Sécurité physique (badges, caméras, gardiens)
  • 💰 Rackage et câblage
  • 💰 Remplacement hardware défaillant
  • 💰 Personnel d'exploitation (24/7 pour production)
  • 💰 Conformité et audits

✅ Avec AWS : Concentrez vos ressources sur votre CŒUR DE MÉTIER et la différenciation business, pas sur l'infrastructure indifférenciée

6️⃣ Déploiement mondial en minutes

Infrastructure mondiale AWS :
  • 🌍 33+ Régions géographiques
  • 🏢 105+ Zones de Disponibilité (AZ)
  • ⚡ 600+ Points de Présence (Edge Locations)
  • 🌐 Présence sur tous les continents (sauf Antarctique)

✅ Avantage :

  • Déployez à Tokyo, São Paulo et Paris en quelques clics
  • Faible latence pour vos utilisateurs mondiaux
  • Conformité réglementaire locale (souveraineté des données)
  • Disaster recovery géographique

🏗️1.2 AWS Well-Architected Framework

Définition : Ensemble de bonnes pratiques pour concevoir et opérer des systèmes fiables, sécurisés, efficaces et rentables dans le cloud.

Les 6 Piliers du Framework

Pilier 1 : Excellence Opérationnelle

Objectif : Exécuter et surveiller les systèmes pour apporter de la valeur métier

Principes clés :

  • Opérations as Code : Infrastructure et procédures en code (IaC)
  • Changements fréquents et petits : Faciles à annuler
  • Anticiper les échecs : Tests de défaillance réguliers
  • Apprendre des échecs : Post-mortems sans blâme
  • Amélioration continue : Mesurer, apprendre, améliorer

Services AWS : CloudFormation, Config, CloudTrail, CloudWatch

📝 Exemple concret : Une équipe configure manuellement chaque nouveau serveur via la console AWS (clic par clic), ce qui prend 2h et génère des erreurs de configuration différentes à chaque fois.
→ Application du pilier : Décrire le serveur dans un template CloudFormation. Chaque déploiement devient identique, reproductible en 5 minutes, et versionné dans Git comme du code source.
Scénario type examen : "Une entreprise veut s'assurer que tous ses environnements (dev, staging, prod) ont une configuration identique." → Réponse attendue : Infrastructure as Code (CloudFormation).

Pilier 2 : Sécurité

Objectif : Protéger informations, systèmes et actifs

Principes clés :

  • Identité forte : Moindre privilège, MFA
  • Traçabilité : Logs centralisés, alertes temps réel
  • Sécurité à tous les niveaux : Défense en profondeur
  • Chiffrement : En transit ET au repos
  • Automatisation : Security baselines
  • Préparation aux incidents : Playbooks, simulations

Services AWS : IAM, KMS, GuardDuty, Security Hub, WAF

📝 Exemple concret : Un développeur reçoit des accès "Administrateur complet" sur AWS juste pour pouvoir lire des logs CloudWatch dans son projet.
→ Application du pilier : Créer une politique IAM qui n'autorise QUE la lecture des logs de ce projet précis (moindre privilège). Si son compte est compromis, l'attaquant ne peut rien faire d'autre que lire ces logs.
Scénario type examen : "Un utilisateur a accidentellement supprimé une base de données car il avait trop de droits." → Le problème vient d'une violation du principe du moindre privilège.

Pilier 3 : Fiabilité

Objectif : Fonctionnement correct et récupération rapide des pannes

Principes clés :

  • Récupération automatique : Self-healing
  • Tester les procédures de récupération : Chaos engineering
  • Mise à l'échelle horizontale : Ajouter des ressources
  • Auto Scaling : Basé sur métriques
  • Changement via automation : Éviter erreurs manuelles

Concepts importants :

  • RTO : Recovery Time Objective (temps max d'interruption)
  • RPO : Recovery Point Objective (perte données max)
  • Multi-AZ : Haute disponibilité
  • Multi-Region : Disaster recovery

Services AWS : Route 53, ELB, Auto Scaling, Backup

📝 Exemple concret : Une application tourne sur une seule instance EC2 dans une seule AZ. Si cette AZ subit une panne (incendie, coupure réseau), le site est down pendant des heures.
→ Application du pilier : Déployer au moins 2 instances dans 2 AZ différentes derrière un Load Balancer + Auto Scaling Group. Si une AZ tombe, l'autre continue de servir le trafic automatiquement.
Scénario type examen : "Une entreprise doit garantir 99.99% de disponibilité même en cas de panne d'un data center entier." → Réponse attendue : architecture Multi-AZ.

Pilier 4 : Efficacité des Performances

Objectif : Utiliser efficacement les ressources informatiques

Principes clés :

  • Démocratiser technologies avancées : ML, databases managées
  • Déploiement mondial : Réduire latence
  • Architectures serverless : Pas de gestion serveurs
  • Expérimenter : Tester différentes configs
  • Affinité mécanique : Bonne techno pour bon usage

5 domaines : Calcul, Stockage, Base de données, Réseau, Revue continue

📝 Exemple concret : Une entreprise paie pour une grosse instance EC2 qui tourne 24/7, mais qui ne traite des requêtes que 10 minutes par jour.
→ Application du pilier : Remplacer par AWS Lambda (serverless) : aucune ressource ne tourne en dehors des 10 minutes d'usage réel, et la facturation suit l'usage exact à la milliseconde.
Scénario type examen : "Une fonction de traitement d'image s'exécute de façon imprévisible, parfois 0 fois par jour, parfois 1000 fois." → Réponse attendue : architecture serverless (Lambda) plutôt que des serveurs EC2 permanents.

Pilier 5 : Optimisation des Coûts

Objectif : Meilleur ROI, éviter dépenses inutiles

Principes clés :

  • Modèle de consommation : Payez ce dont vous avez besoin
  • Mesurer efficacité : Métriques métier
  • Services managés : Réduire travail indifférencié
  • Analyser dépenses : Tags, allocation par projet
  • Right-sizing : Adapter taille à utilisation réelle

Stratégies :

  • Reserved Instances : -72% sur 3 ans
  • Spot Instances : -90% pour charges interruptibles
  • Savings Plans : Flexibilité + économies
  • Arrêter ressources inutilisées
📝 Exemple concret : Une entreprise a 50 instances On-Demand qui tournent en permanence depuis 2 ans pour une charge stable et prévisible — elle paie donc le prix plein, jour après jour.
→ Application du pilier : Acheter des Reserved Instances ou un Savings Plan d'1 à 3 ans pour cette charge stable connue, réduisant la facture jusqu'à 72% sans rien changer techniquement.
Scénario type examen : "Une entreprise utilise des serveurs de batch tolérants aux interruptions, qui peuvent redémarrer si coupés." → Réponse attendue : Spot Instances (le scénario "tolérant aux interruptions" est le signal-clé).

Pilier 6 : Durabilité (Nouveau 2022)

Objectif : Minimiser impact environnemental

Principes clés :

  • Comprendre impact : Métriques carbone
  • Objectifs durabilité : Réduction empreinte
  • Maximiser utilisation : Taux élevé des ressources
  • Nouvelles technologies : Hardware plus efficace
  • Services managés : Mutualisation
  • Réduire impact aval : Compression, caching

🚚1.3 Stratégies de Migration - Les 7 R

Définition : Les "7 R" décrivent les 7 stratégies possibles pour migrer une application existante vers AWS, du plus simple/rapide au plus transformateur.
Stratégie Description Effort Exemple
1. Rehost
(Lift and Shift)		 Déplacer l'application telle quelle vers AWS, sans aucune modification du code 🟢 Faible Migrer un serveur on-premises vers une instance EC2 équivalente
2. Replatform
(Lift, Tinker and Shift)		 Quelques optimisations ciblées pendant la migration, sans changer l'architecture globale 🟡 Moyen Migrer une base de données on-premises vers Amazon RDS (le moteur reste identique, mais AWS gère désormais l'infrastructure)
3. Repurchase
(Drop and Shop)		 Abandonner l'ancienne solution et racheter une nouvelle solution SaaS 🟡 Moyen Remplacer un CRM auto-hébergé par Salesforce
4. Refactor
(Re-architect)		 Repenser entièrement l'architecture de l'application pour exploiter pleinement le cloud (microservices, serverless...) 🔴 Élevé Transformer une application monolithique en architecture microservices avec Lambda et DynamoDB
5. Retire Identifier et supprimer des applications devenues inutiles, ne pas les migrer du tout 🟢 Aucun Arrêter un outil interne que plus personne n'utilise
6. Retain
(Revisit)		 Garder l'application on-premises pour le moment (pas encore prête à migrer, contrainte réglementaire...) ⚪ Aucun Conserver une application legacy critique tant qu'elle n'est pas documentée
7. Relocate Déplacer l'infrastructure (souvent des VM VMware) vers AWS sans changer d'outils de gestion ni racheter du matériel 🟢 Faible Migrer un parc VMware vers VMware Cloud on AWS
💡 À retenir pour l'examen :
  • Rehost = le plus rapide, le plus courant en première vague de migration
  • Refactor = le plus long et coûteux, mais le plus grand bénéfice cloud à terme
  • Retire et Retain = les deux seules stratégies où l'application ne va PAS sur AWS (immédiatement)
  • Une question scénario décrit souvent une contrainte (budget, temps, besoin de modernisation) → identifiez quelle stratégie correspond

🏗️1.4 AWS Cloud Adoption Framework (CAF)

Définition : Le AWS Cloud Adoption Framework (CAF) est un guide structuré pour aider les organisations à planifier et exécuter leur migration vers le cloud. Il identifie les compétences et processus nécessaires pour une adoption cloud réussie.

Les 6 Perspectives du AWS CAF

Le CAF organise les capacités en 6 perspectives réparties en 2 catégories :

📊 Capacités Métier

Ces perspectives se concentrent sur les aspects organisationnels et de gouvernance.

1️⃣ Perspective Métier (Business)
  • Objectif : Aligner les investissements cloud avec les objectifs business
  • Rôles clés : CEO, CFO, COO, directeurs métier
  • Capacités :
    • Gestion de portefeuille de projets cloud
    • Création de business case pour le cloud
    • Mesure du ROI et des bénéfices
    • Gestion du changement organisationnel
2️⃣ Perspective Personnes (People)
  • Objectif : Préparer les équipes aux changements culturels et organisationnels
  • Rôles clés : DRH, responsables formation, gestionnaires d'équipe
  • Capacités :
    • Formation et développement des compétences cloud
    • Gestion de la conduite du changement
    • Nouvelle structure organisationnelle (DevOps, équipes cloud)
    • Culture d'innovation et d'expérimentation
3️⃣ Perspective Gouvernance (Governance)
  • Objectif : Orchestrer les initiatives cloud tout en minimisant les risques
  • Rôles clés : CIO, responsables PMO, architectes d'entreprise, risk managers
  • Capacités :
    • Gestion du portefeuille de projets
    • Gouvernance des coûts et budgets cloud
    • Gestion des risques et conformité
    • Métriques et KPI cloud

⚙️ Capacités Techniques

Ces perspectives couvrent les aspects technologiques de l'adoption cloud.

4️⃣ Perspective Plateforme (Platform)
  • Objectif : Construire une plateforme cloud scalable et moderne
  • Rôles clés : CTO, architectes techniques, ingénieurs infrastructure
  • Capacités :
    • Architecture cloud hybride et multi-compte
    • Design de réseaux VPC, connectivité (Direct Connect, VPN)
    • Stratégie compute (EC2, containers, serverless)
    • Services managés et modernisation d'applications
5️⃣ Perspective Sécurité (Security)
  • Objectif : Garantir confidentialité, intégrité et disponibilité des données
  • Rôles clés : CISO, responsables sécurité, analystes conformité
  • Capacités :
    • Gestion des identités et accès (IAM)
    • Détection et réponse aux incidents
    • Protection des données (chiffrement, DLP)
    • Conformité et audits (PCI-DSS, HIPAA, GDPR)
6️⃣ Perspective Opérations (Operations)
  • Objectif : Exploiter et monitorer les workloads cloud de manière fiable
  • Rôles clés : Responsables IT, SRE, équipes DevOps, support
  • Capacités :
    • Monitoring et observabilité (CloudWatch, X-Ray)
    • Automatisation des opérations (runbooks, SSM)
    • Gestion des incidents et service management
    • Performance et optimisation continue
📝 Exemple concret d'utilisation du CAF :

Une banque veut migrer vers AWS. Le CAF structure le projet :

  • Business : Justifier le ROI de la migration (réduction coûts datacenter 30%)
  • People : Former 50 développeurs aux certifications AWS
  • Governance : Mettre en place FinOps pour contrôler les dépenses cloud
  • Platform : Concevoir architecture multi-compte avec AWS Organizations
  • Security : Implémenter conformité PCI-DSS avec AWS Artifact + GuardDuty
  • Operations : Déployer monitoring centralisé avec CloudWatch + SNS
🎯 Pour l'examen CLF-C02 :
  • ✅ Connaître les 6 perspectives et leur objectif principal
  • ✅ Distinguer perspectives Métier (3) vs Techniques (3)
  • ✅ Associer les bons rôles à chaque perspective
  • ✅ Comprendre que le CAF est un framework d'ADOPTION, pas une architecture technique
  • ❌ Pas besoin de mémoriser toutes les capacités détaillées
💡 Pour l'examen :
  • Mémorisez les 6 avantages du cloud par cœur
  • Connaissez les 6 piliers du Well-Architected Framework
  • Connaissez les 6 perspectives du CAF (3 Métier + 3 Techniques)
  • Associez services AWS à chaque pilier
  • Comprenez QUAND appliquer chaque principe

🔒 Module 2: Sécurité et Conformité (30% de l'examen)

🔴 LE PLUS IMPORTANT : Ce module représente 30% de l'examen (~15 questions). La sécurité est au cœur d'AWS. Maîtrisez parfaitement le modèle de responsabilité partagée et IAM.

🤝2.1 Modèle de Responsabilité Partagée

Principe fondamental :

AWS = Sécurité DU cloud
Client = Sécurité DANS le cloud

RESPONSABILITÉS AWS (Sécurité DU Cloud)
Infrastructure physique Datacenters, sécurité périmétrique, accès biométrique, gardiennage 24/7, caméras, détection intrusion
Hardware & Réseau Serveurs, stockage, routeurs, switches, câblage
Hyperviseur Virtualisation, isolation entre clients, nettoyage mémoire
Services managés Patching RDS/Aurora, scaling DynamoDB, gestion Lambda
RESPONSABILITÉS CLIENT (Sécurité DANS le Cloud)
Données Classification, chiffrement, sauvegarde, durée rétention
IAM Users, groups, roles, policies, MFA, rotation clés
OS (EC2) Patching, firewall, antivirus, configuration
Applications Code, vulnérabilités (SQL injection, XSS), validation entrées
Réseau VPC Security Groups, NACL, subnets, VPN
Chiffrement Activation S3/EBS/RDS, gestion clés KMS

Le Modèle Varie selon le Service

📊 IaaS (Infrastructure) - EC2 :
  • ✅ AWS : Hardware, hyperviseur, réseau physique
  • ❌ VOUS : OS, applications, données, firewall, patching, chiffrement
  • 🔴 Responsabilité CLIENT ÉLEVÉE
📦 Conteneurs - ECS, Fargate :
  • ✅ AWS : Infrastructure, orchestration (Fargate = tout)
  • ❌ VOUS : Configuration conteneurs, images, sécurité applicative
  • 🟡 Responsabilité CLIENT MOYENNE
🚀 Services Managés - S3, DynamoDB, Lambda, RDS :
  • ✅ AWS : Infrastructure, OS, patching, scaling, disponibilité
  • ❌ VOUS : Données, politiques d'accès, activation chiffrement
  • 🟢 Responsabilité CLIENT FAIBLE
🎯 Pour l'examen - Règles simples :
  • ✅ Si c'est PHYSIQUE → AWS
  • ✅ Si vous pouvez le CONFIGURER → VOUS
  • ✅ Plus le service est MANAGÉ → Plus AWS en fait
  • ✅ EC2 : Vous gérez presque tout sauf hardware
  • ✅ RDS : AWS patch l'OS et la DB, vous gérez les données
  • ✅ Lambda/S3 : AWS gère presque tout, vous gérez données et accès

👤2.2 IAM - Guide Complet

Service GLOBAL : IAM n'est PAS régional. Users/Groups/Roles disponibles dans TOUTES les régions.

Composant 1 : IAM Users (Utilisateurs)

Définition : Identité permanente pour une personne ou application

Types d'accès :

  • Console AWS : Username + Password + MFA (recommandé)
  • Programmatique (CLI/SDK) : Access Key ID + Secret Access Key

Limites AWS :

  • Max 5000 users par compte
  • Max 2 access keys par user
  • Un user dans max 10 groups
🔴 Bonnes pratiques CRITIQUES :
  • ❌ JAMAIS partager credentials IAM
  • ✅ Rotation access keys tous les 90 jours
  • ✅ Désactiver keys inutilisées (AWS alerte si > 90j)
  • ✅ Utiliser ROLES pour EC2 au lieu d'access keys
  • ✅ MFA pour tous les users console

Composant 2 : IAM Groups (Groupes)

Définition : Collection d'utilisateurs partageant les mêmes permissions

Caractéristiques :

  • ❌ Les groupes NE PEUVENT PAS être imbriqués
  • ✅ Un user peut appartenir à plusieurs groups
  • ✅ Permissions héritées de TOUS les groups
  • Max 300 groups par compte

Exemples typiques :

  • Admins : AdministratorAccess (accès complet)
  • Developers : EC2, Lambda, DynamoDB, S3
  • DataScientists : SageMaker, S3, Athena, Glue
  • ReadOnly : ViewOnlyAccess (consultation)

Composant 3 : IAM Roles (Rôles)

Définition : Identité avec permissions spécifiques mais SANS credentials permanents. Credentials temporaires auto-renouvelés.

Cas d'usage principaux :

A. Rôles pour services AWS (LE PLUS COURANT) :
  • EC2 Instance Role : Permet à EC2 d'appeler APIs AWS sans access keys
    • Exemple : EC2 → S3, EC2 → DynamoDB
  • Lambda Execution Role : Permissions pour Lambda (logs CloudWatch, S3, etc.)
  • ECS Task Role : Permissions pour conteneurs ECS
B. Rôles inter-comptes (Cross-Account) :
  • Compte A accède aux ressources du Compte B
  • Usage : Organisations multi-comptes
C. Rôles pour fédération :
  • Active Directory on-premise → AWS
  • SAML 2.0, OpenID Connect
  • AWS IAM Identity Center (SSO)

Durée credentials : Défaut 1h, configurable 15min à 12h max

Composant 4 : IAM Policies (Politiques)

Définition : Document JSON définissant permissions (Allow ou Deny) 
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": ["s3:GetObject", "s3:PutObject"],
      "Resource": "arn:aws:s3:::mon-bucket/*",
      "Condition": {
        "IpAddress": {
          "aws:SourceIp": "203.0.113.0/24"
        }
      }
    }
  ]
}

Types de policies :

  • AWS Managed : Créées par AWS, mises à jour auto
    • AdministratorAccess : Tout
    • PowerUserAccess : Tout sauf IAM
    • ReadOnlyAccess : Lecture seule
  • Customer Managed : Créées par vous, réutilisables
  • Inline : Attachées à 1 user/group/role, non réutilisables

🔴 Compte Root AWS - ULTRA CRITIQUE

Le compte root a un accès ILLIMITÉ :
  • Peut TOUT faire sans restriction
  • Peut supprimer le compte AWS
  • Aucune policy IAM ne peut le restreindre

✅ Bonnes pratiques OBLIGATOIRES :

  1. 🔴 Activer MFA IMMÉDIATEMENT (hardware token recommandé)
  2. 🔴 NE JAMAIS l'utiliser au quotidien
  3. 🔴 Créer un user IAM admin pour travail quotidien
  4. 🔴 Sécuriser l'email root (2FA sur email)
  5. 🔴 NE JAMAIS créer d'access keys pour root
  6. 🔴 Mot de passe complexe unique (gestionnaire MDP)
💡 Pour l'examen :
  • 📌 User = credentials permanents, Role = credentials temporaires
  • 📌 Toujours utiliser Roles pour applications EC2
  • 📌 MFA obligatoire sur compte root
  • 📌 Principe du moindre privilège TOUJOURS
  • 📌 Groupes NE peuvent PAS être imbriqués

🌐 Amazon VPC (Virtual Private Cloud) - Guide Technique Complet

Réseau virtuel logiquement isolé dans le cloud AWS. Contrôle total sur l'environnement réseau virtuel.

📌VPC par Défaut

VPC par défaut créé automatiquement dans chaque région AWS
  • CIDR: 172.31.0.0/16 (65536 adresses IP)
  • Un subnet public par défaut dans chaque AZ
  • Internet Gateway attachée automatiquement
  • Route table par défaut avec route vers Internet
  • Security group par défaut (permet tout le trafic sortant)
  • Network ACL par défaut (permet tout)
  • Peut être supprimé et recréé si nécessaire
⚠️ Attention : Ne PAS utiliser le VPC par défaut en production pour des raisons de sécurité

🔧Composants VPC - Détails Techniques

NAT Gateway

Service managé AWS permettant aux instances des subnets privés d'accéder à Internet (sortant uniquement)

Caractéristiques :

  • Service managé (pas de gestion de serveur)
  • Haute disponibilité dans une AZ
  • Scalabilité automatique jusqu'à 100 Gbps
  • Supporte 55000 connexions simultanées vers une destination unique
  • Supporte TCP, UDP, ICMP

Tarification :

  • Par heure : 0,045 $/heure (~32 $/mois par NAT Gateway)
  • Traitement données : 0,045 $/GB de données traitées
  • 💡 1 NAT Gateway par AZ pour éviter frais inter-AZ
🔴 Haute Disponibilité : Créer 1 NAT Gateway par AZ pour éviter le single point of failure

Security Groups

Pare-feu virtuel au niveau de l'instance EC2/ENI

Stateful (si trafic entrant autorisé, retour automatiquement autorisé)

Caractéristiques :

  • Contrôle le trafic au niveau instance (ENI)
  • Règles ALLOW uniquement (pas de DENY explicite)
  • Évalue toutes les règles avant de décider
  • Supporte règles par IP, CIDR, ou autre Security Group
  • Changements appliqués immédiatement

💾 Services de Stockage

AWS propose 3 grandes familles de stockage : objet (S3), bloc (EBS), fichier (EFS). Savoir laquelle choisir selon le scénario est une question fréquente à l'examen.

📦Amazon S3 (Simple Storage Service)

Stockage objet : illimité, accessible via API/HTTP, organisé en buckets (noms uniques globalement) contenant des objets (jusqu'à 5 To chacun).
  • Durabilité : 99.999999999% (11 "9") sur toutes les classes sauf One Zone-IA
  • Versioning : conserve plusieurs versions d'un même objet
  • Lifecycle policies : transition automatique entre classes de stockage (ex: Standard → Glacier après 90 jours)
  • S3 Static Website Hosting : héberger un site statique directement depuis un bucket
💡 À retenir : Les classes de stockage S3 (Standard, IA, Glacier...) sont détaillées dans l'onglet 📊 Comparaisons.

💿Amazon EBS (Elastic Block Store)

Stockage bloc attaché à UNE seule instance EC2 à la fois (sauf io2 Multi-Attach), dans UNE seule Availability Zone.
  • Persiste indépendamment du cycle de vie de l'instance EC2
  • Types : gp3/gp2 (SSD usage général), io2/io1 (SSD haute performance IOPS), st1 (HDD optimisé débit), sc1 (HDD froid, le moins cher)
  • Snapshots : sauvegardes incrémentales stockées sur S3, utilisables pour créer de nouveaux volumes ou migrer entre AZ

📁Amazon EFS (Elastic File System)

Stockage fichier (protocole NFS) qui peut être monté simultanément sur PLUSIEURS instances EC2, dans PLUSIEURS Availability Zones.
  • Capacité élastique automatique (pas de provisionnement de taille)
  • Cas d'usage : partage de fichiers entre serveurs web, environnements de conteneurs partagés

🗄️ Services de Base de Données

L'examen teste votre capacité à choisir le BON type de base de données selon le besoin (relationnel vs NoSQL, charge en lecture vs écriture, etc.).

🔗Amazon RDS (Relational Database Service)

Base de données relationnelle managée : AWS gère le patching, les sauvegardes, le failover.
  • Moteurs supportés : MySQL, PostgreSQL, MariaDB, Oracle, SQL Server
  • Multi-AZ : réplica synchrone dans une autre AZ pour la haute disponibilité (failover automatique)
  • Read Replicas : réplicas asynchrones pour répartir la charge de LECTURE (pas pour la haute dispo)

⚡Amazon Aurora

Base relationnelle compatible MySQL/PostgreSQL, conçue par AWS pour des performances supérieures (jusqu'à 5x MySQL).
  • Stockage distribué auto-scalable, jusqu'à 15 réplicas en lecture
  • Aurora Serverless : capacité qui s'ajuste automatiquement selon la charge

🔑Amazon DynamoDB

Base NoSQL clé-valeur / document, entièrement managée et serverless.
  • Mise à l'échelle horizontale automatique, latence à un chiffre en millisecondes
  • Modes de capacité : à la demande (pay-per-request) ou provisionnée
  • DynamoDB Accelerator (DAX) : cache en mémoire pour latence micro-seconde

🚀Amazon ElastiCache

Cache en mémoire managé (Redis ou Memcached), placé devant une base de données pour réduire la latence des lectures fréquentes.

📊Amazon Redshift

Entrepôt de données (data warehouse) pour l'analytique à grande échelle (requêtes SQL sur des pétaoctets de données), différent d'une base transactionnelle classique.

🐳 Services de Conteneurisation

Les conteneurs permettent d'empaqueter une application avec toutes ses dépendances dans une unité portable et légère, garantissant qu'elle fonctionne de manière identique quel que soit l'environnement.

💡 Pourquoi les conteneurs ?
  • ✅ Portabilité : "Build once, run anywhere" - fonctionne en dev, test, prod
  • ✅ Isolation : Chaque conteneur est isolé (dépendances, environnement)
  • ✅ Légèreté : Plus léger qu'une VM (pas d'OS complet, démarre en secondes)
  • ✅ Scalabilité : Lance des centaines de conteneurs rapidement
  • ✅ Microservices : Idéal pour architectures modernes découplées

📦Amazon Elastic Container Service (ECS)

Service d'orchestration de conteneurs natif AWS (propriétaire)

Caractéristiques :

  • Orchestration de conteneurs Docker entièrement managée par AWS
  • Intégration native profonde avec les services AWS (ALB, CloudWatch, IAM, Secrets Manager)
  • Pas de control plane à gérer (AWS gère l'orchestration)
  • Moins complexe à apprendre que Kubernetes pour débuter

Deux modes de lancement :

Mode Description Gestion Cas d'usage
ECS sur EC2 Conteneurs tournent sur des instances EC2 que VOUS gérez ❌ Vous gérez les instances (patching, scaling, capacité) Besoin de contrôle total, optimisation coûts avancée, charges stables
ECS sur Fargate Conteneurs tournent sans serveurs (serverless) ✅ AWS gère tout (pas d'instances visibles) Déploiement rapide, charges variables, focus sur l'application
📝 Exemple : Une startup veut déployer une application conteneurisée sans gérer de serveurs → ECS sur Fargate (serverless, simple, rapide à démarrer).

☸️Amazon Elastic Kubernetes Service (EKS)

Service Kubernetes managé par AWS

Kubernetes (K8s) : Système d'orchestration de conteneurs open-source créé par Google, devenu le standard industriel. Plus complexe mais plus puissant et portable.

Caractéristiques :

  • Kubernetes 100% standard (pas de lock-in AWS)
  • AWS gère le control plane (master nodes) → haute disponibilité automatique
  • Compatible avec outils Kubernetes existants (kubectl, Helm, Prometheus)
  • Portabilité multi-cloud (K8s fonctionne sur AWS, Azure, GCP, on-premises)
  • Écosystème riche : CNCF (Cloud Native Computing Foundation)

Deux modes de lancement également :

  • EKS sur EC2 : Worker nodes sur instances EC2 (vous gérez)
  • EKS sur Fargate : Worker nodes serverless (AWS gère)
📝 Exemple : Une entreprise a déjà des applications Kubernetes on-premises et veut migrer vers AWS tout en gardant la même plateforme → EKS (compatibilité K8s standard).

🚀AWS Fargate (Serverless pour conteneurs)

Moteur de calcul serverless pour conteneurs (fonctionne avec ECS ET EKS)

Concept clé : Fargate n'est PAS un service d'orchestration. C'est un mode de lancement qui retire la gestion des serveurs.

Avantages :

  • ✅ Zéro gestion de serveurs : pas d'EC2, pas de patching, pas de scaling d'instances
  • ✅ Facturation à la ressource (vCPU + RAM utilisés par conteneur) et à la seconde
  • ✅ Scaling automatique : chaque tâche/pod démarre dans son environnement isolé
  • ✅ Sécurité renforcée : isolation au niveau kernel pour chaque conteneur

Quand utiliser Fargate :

  • 🎯 Workloads variables (pics de trafic imprévisibles)
  • 🎯 Équipes petites (pas de DevOps dédiés pour gérer EC2)
  • 🎯 Microservices avec scaling indépendant
  • 🎯 Prototypes / startups (time-to-market rapide)

Quand NE PAS utiliser Fargate :

  • ❌ Besoin d'accès aux instances sous-jacentes (GPUs, instance store)
  • ❌ Workloads très stables 24/7 (EC2 Reserved Instances plus économiques)
  • ❌ Besoins réseau avancés (ENI dédiée, placement groups)

📦Amazon Elastic Container Registry (ECR)

Registry Docker privé managé par AWS (équivalent AWS de Docker Hub)

Rôle : Stocker, gérer et déployer des images de conteneurs Docker

Caractéristiques :

  • Intégration IAM : contrôle d'accès granulaire aux images
  • Scan de vulnérabilités automatique (détection CVE)
  • Chiffrement des images au repos (KMS)
  • Réplication multi-région et cross-account
  • Fonctionne avec ECS, EKS, mais aussi on-premises (via pull)
📝 Workflow typique :
  1. Développeur build une image Docker localement
  2. docker push vers ECR (stockage privé sécurisé)
  3. ECS/EKS pull l'image depuis ECR pour lancer le conteneur
  4. ECR scanne l'image pour détecter les vulnérabilités

🗄️Bases de Données : RDS vs Aurora vs DynamoDB

Critère RDS Aurora DynamoDB
Type Relationnel (SQL) Relationnel (SQL), compatible MySQL/PostgreSQL NoSQL (clé-valeur / document)
Mise à l'échelle Verticale (changer de taille d'instance) Stockage auto-scalable, jusqu'à 15 réplicas Horizontale automatique, sans limite pratique
Performance Standard Jusqu'à 5x MySQL, 3x PostgreSQL Latence à un chiffre en millisecondes
Cas d'usage Apps traditionnelles, ERP, CRM Apps critiques nécessitant haute dispo + perf SQL Apps web/mobile à très grande échelle, IoT, gaming

🧮Compute : EC2 vs Lambda vs ECS/Fargate

Critère EC2 Lambda ECS / Fargate
Modèle Serveur virtuel complet Fonctions sans serveur (event-driven) Conteneurs (Docker)
Gestion infra Vous gérez l'OS, patching, scaling Zéro gestion serveur Fargate = zéro serveur ; ECS sur EC2 = vous gérez les instances
Durée d'exécution Illimitée (tant que l'instance tourne) Max 15 minutes par invocation Illimitée
Facturation Par seconde tant que l'instance tourne Par invocation + durée d'exécution (ms) Par ressources (vCPU/mémoire) allouées

🔐Security Group vs Network ACL

Critère Security Group Network ACL
Niveau Instance (ENI) Subnet
État Stateful (réponse autorisée automatiquement) Stateless (règles entrée ET sortie nécessaires)
Règles Autorisation uniquement (Allow only) Autorisation ET refus explicite (Allow + Deny)
Évaluation Toutes les règles évaluées avant décision Règles évaluées par ordre de numéro, la 1ère qui correspond s'applique
Limite par défaut 5 SG par interface réseau (extensible) 1 NACL par subnet (peut être partagée)

👤IAM : Users vs Groups vs Roles

Critère User Group Role
Représente Une personne ou application identifiée Un ensemble d'Users Une identité temporaire, assumable
Identifiants Permanents (mot de passe / clés d'accès) N/A (hérite des Users membres) Temporaires (via STS, expirent)
Cas d'usage type Accès individuel à la console/API Gérer les permissions de plusieurs Users en une fois Service AWS accédant à d'autres services (ex: EC2 → S3), accès cross-account, fédération
⚠️ Bonne pratique AWS : Toujours préférer un Role à des clés d'accès permanentes lorsqu'un service AWS doit accéder à un autre service.
